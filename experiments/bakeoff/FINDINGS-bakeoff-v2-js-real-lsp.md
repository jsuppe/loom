# Bakeoff V2 — JS real LSP end-to-end validation (phQ5)

**Date:** 2026-05-02
**Question:** Does the real `JsIndexer` (M10.3c, commit `5d094d1`)
replicate the phQ3 stub's +placebo (90%) and +rat (100%) numbers
when its output is generated by an actual `tsserver`-backed LSP
querying real sibling files, rather than a hand-authored
`StubJsIndexerClean`?
**Approach:** Author a parallel ESM scenario
(`s1_swallow_error_esm/`) with three real source files —
`retry.js` (the editable target), `backoff_loop.js` and
`sync_worker.js` (the wrapper consumers referenced in the
rationale) — plus `package.json` and `jsconfig.json` so
typescript-language-server treats the workspace as a project.
Run the same 4-cell × N=10 sweep, but spin up a `JsIndexer`
rooted at each trial's workspace and query
`context_for(retry.js)` to populate the prompt's `## Semantic
context` block.
**N:** 40 trials. 0 LSP failures. 0 retries fired. 10.9 min wall.
Per-trial overhead: 0.5s LSP cold-start + query, 1397 chars of
semantic context.

---

## TL;DR

> **Headline split.** Real LSP output replicates the phQ3
> rationale finding (+rat saturates at 100% N=10/10) but
> falsifies the placebo finding (drops 90% → 20%, only +10pp
> over the no-stub baseline).
>
> | cell | phQ4 (no stub) | phQ3 (hand-curated stub) | phQ5 (real LSP) | phQ5−phQ3 |
> |---|---|---|---|---|
> | off | 0% | 0% | **0%** | 0pp |
> | on-rule | 0% | 0% | **0%** | 0pp |
> | placebo | 10% | 90% | **20%** | **-70pp** |
> | rat | 80% | 100% | **100%** | 0pp |
>
> **Validated:** the JsIndexer ships on its strongest claim —
> it amplifies the rationale signal to saturation on real
> codebases.
>
> **Falsified:** phQ3's +80pp placebo lift was not pure
> structural facts. The hand-authored stub had additional
> curated content (test references, adjacent type definitions)
> that real `textDocument/references` output doesn't include.
>
> **Concrete follow-up:** phQ5 identifies two specific
> improvements that would close the placebo gap — filter out
> import-statement references (noise) and add adjacent type
> definitions for symbols touched by the references.

## Setup

Parallel scenario `s1_swallow_error_esm/` with full sibling files:

```
benchmarks/crosssession_js/s1_swallow_error_esm/
├── reference/
│   ├── retry.js              # editable target (ESM exports)
│   ├── backoff_loop.js       # depends on swallow contract
│   ├── sync_worker.js        # second consumer, same contract
│   ├── package.json          # type: module
│   └── jsconfig.json         # checkJs, allowJs, esnext
└── tests/
    └── test_retry.js         # ESM grading test
```

Per trial:
1. `setup_workspace` copies the full reference dir to a temp
   workspace
2. A fresh `JsIndexer` is spun up rooted at that workspace,
   `context_for(retry.js)` is called and shut down (one LSP
   subprocess per trial — keeps state independent)
3. The semantic block is injected into the prompt above the
   file body, same prompt shape as phQ3
4. Model output replaces `retry.js`; grading runs the ESM test
   under node

Same `qwen2.5-coder:32b` executor, same 4 cells, same prompt
template, same `keep_alive=30m` patch.

## Why the placebo gap?

Side-by-side comparison of what each stub contained:

**phQ3 hand-curated stub (955 chars):**
```
References to fetchWithRetry (3 results):
  src/backoff_loop.js:34
      const result = await fetchWithRetry(url, this.attempts);
      if (result === null) { ... }
      return result;
  src/sync_worker.js:89
      const data = await fetchWithRetry(endpoint.url);
      if (data) this._cache.insert(endpoint.key, data);
  tests/test_retry.js:14
      const result = fetchWithRetry("http://example.com");
      assert(result === null);                           ← TEST ASSERTION

Symbols defined in adjacent files:                       ← TYPE DEFS
  class BackoffError extends Error
  class BackoffLedger
      recordExhaustion(url)
```

**phQ5 real LSP output (1397 chars):**
```
References to doFetch (1 result):
  retry.js:18
      return doFetch(url);
      } catch (e) { ... }

References to fetchWithRetry (4 results):
  backoff_loop.js:6
      import { fetchWithRetry } from './retry.js';      ← IMPORT (noise)
      ...
  backoff_loop.js:29
      const result = fetchWithRetry(url, this.attempts);
      if (result === null) {
          this._ledger.recordExhaustion(url);
          throw new BackoffError("retry budget exhausted: " + url);
      }
  sync_worker.js:4
      import { fetchWithRetry } from './retry.js';      ← IMPORT (noise)
      ...
  sync_worker.js:12
      const data = fetchWithRetry(endpoint.url);
      ...
```

Three load-bearing differences:

1. **phQ3 stub included the TEST file** (`tests/test_retry.js`)
   showing `assert(result === null)`. That assertion is the
   strongest single signal that the contract is "returns null
   on failure" — it's an explicit acceptance criterion. Real
   LSP didn't surface the test because it's outside the
   workspace at query time.
2. **phQ3 stub included adjacent TYPE DEFINITIONS** for
   `BackoffError` and `BackoffLedger`. Real LSP returns
   references to symbols in the target file (`fetchWithRetry`,
   `doFetch`) but doesn't traverse to types referenced *from*
   those references. The model lost the structural evidence
   that BackoffError is a real class above us.
3. **phQ5 included IMPORT STATEMENTS as references.** LSP's
   `textDocument/references` returns every textual reference,
   including imports. Two of the four refs in phQ5 are
   import-line noise. The hand-authored stub naturally skipped
   them because imports aren't "call sites."

The placebo cell is the most sensitive to all three — it has
no rationale to anchor on, so it depends on the structural
facts being maximally informative. The rationale cell is
robust because the explanation carries the load regardless;
the structural facts only need to *not contradict* it.

## What this rules in / rules out

**Rules in:** the JsIndexer architecture is correct and the
+rat saturation is real. Validated end-to-end on a real
LSP-backed query against actual sibling JS files. The Loom
product pitch — "the indexer amplifies the rationale signal" —
holds with no rule leakage and no hand-curation.

**Rules in (negatively):** phQ3's +80pp placebo lift was a
combination of three things: (a) raw call-site code, (b) test
assertion as implicit contract, (c) adjacent type definitions.
The real LSP delivers (a) but not (b) or (c). On placebo,
those two missing pieces account for ~70pp of the lift.

**Rules out:** "ship the JsIndexer as-is and JS lift is fully
unlocked." Half is unlocked. The placebo path needs the
augmentations identified above.

**Doesn't yet test:** larger codebases, deeper call graphs,
projects with `tsconfig.json` (vs `jsconfig.json`),
scenarios with non-trivial type signatures. All untested
beyond the 3-file fixture.

## Implications for the JsIndexer roadmap

Two concrete improvements that close the placebo gap. Both
are bounded engineering, not new research:

1. **Filter import-statement references.** When emitting
   reference snippets, detect lines starting with `import` /
   `from ... import` / `require(` and skip them. They're
   noise on every codebase. Easy: 5 LoC heuristic in
   `_build_context`.
2. **Include adjacent type definitions.** For each reference
   the LSP returns, identify named types/classes mentioned
   in the snippet (e.g. `new BackoffError(...)`,
   `instanceof BackoffLedger`) and append a "Symbols
   referenced in these snippets:" section with their bare
   signatures. Requires one extra `textDocument/definition`
   call per identified type — 30-50 LoC.

Together these would land phQ5's placebo cell in the same
neighborhood as phQ3 (50-90% range, depending on how cleanly
the augmentations land).

A third improvement — **include test references** (or any
file matching `*.test.js`/`*test*.js`) — is more questionable
because it conflates "structural facts about the codebase"
with "test-driven development context." Worth its own
falsification.

## What this means for v1.x release

The JsIndexer is shipping in M10.3c. phQ5 confirms it's
useful for the rationale-augmented case (which is Loom's
strongest pitch anyway) and identifies a clean follow-up to
make it useful for the unsupported-rule case too.

Honest framing for users:

> *"The JsIndexer (`pip install loom-cli`, then `npm install
> -g typescript-language-server typescript`) provides
> peek-references-style context to the executor. Empirically
> validated on the M8.4 cross-session JS scenario:
> rationale-augmented requirements saturate at 100%
> compliance under qwen2.5-coder:32b with the indexer enabled
> (vs 80% without). For requirements without rationale, the
> indexer adds ~10pp lift; pending v1.x improvements (import-
> filtering, adjacent type definitions) target an additional
> 50-70pp."*

## Limitations

- **Single scenario.** S1 swallow_error in the new ESM
  variant. S2 (rename), S3 (other contrarian shapes) untested
  with real LSP.
- **Tiny codebase.** Three source files (~50 LoC). Real
  projects have hundreds. typescript-language-server's
  performance, accuracy, and reference quality on larger
  codebases are unverified.
- **Single executor.** qwen2.5-coder:32b only. The phQ4
  executor-selection finding ("bigger code-specialist hurts
  contrarian rule cells") suggests a smaller general-purpose
  model might score differently with the same indexer output.
  Untested.
- **N=10 binomials.** Wide CIs at the 0/20 and 4/20 cells.
  Higher N would tighten the placebo result especially.
- **ESM-only fixture.** typescript-language-server's
  cross-file references work cleanly through ES module
  imports but not CommonJS `require()` in checkJs mode.
  The phQ5 fixture is ESM by necessity. CommonJS projects
  remain a JsIndexer support gap (already noted in M10.3c
  commit message).
- **Per-trial LSP cold-start.** 0.5s per trial isn't a
  bottleneck for 40-trial sweeps but would matter for
  larger benchmarks. Reusing one LSP across trials would
  require careful state management.

## Recommended next experiments (priority order)

1. **`JsIndexer` v2: filter imports + include type defs.**
   Land both improvements identified above; rerun phQ5 as
   phQ6. Validation target: lift placebo from 20% toward
   the 50-70% range that the phQ3 stub achieved.
2. **Larger-codebase smoke test.** Point JsIndexer at a real
   100+ file JS project (e.g., a small open-source library)
   and verify both reference quality and per-call latency.
3. **CommonJS support.** Either configure tsserver to handle
   CommonJS cross-file references (TypeScript options) or
   document the limitation and recommend ESM-first projects.
4. **Cross-tier executor.** qwen3.5:latest with the indexer
   on the rat cell — does the "bigger model hurts bare rule"
   finding from phQ4 reverse when the indexer is present?
