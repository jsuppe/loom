# Bakeoff V2 R6v — Auto-Decomposer Validation (the production pipeline)

**Date:** 2026-04-30
**Question:** Does the production claim from R6m hold when tasks
are produced by `loom decompose --apply` rather than hand-authored?
**Approach:** Same R6 substrate, same chain mechanism, but tasks
generated by `loom decompose` (using `ollama:qwen2.5-coder:32b`,
the local default decomposer). 3 cells × N=20 = 60 trials, 5
metrics, identical to phU R6m for direct comparison.
**N:** 60 trials, 0 harness errors.

---

## TL;DR

> **The production pipeline is validated.** With actual
> auto-decomposed tasks delivered through the standard
> `task_build_prompt` path (D3d), multi-file refactors complete
> at **100 % across all 5 metrics** — identical to R6m's
> hand-authored result. The chain mechanism resolves cross-file
> wiring; the spec context fills in the structural shape that
> auto-decomposed task titles don't carry.
>
> **The R6m D1m/D2m surprise (95–100 % without spec) was an
> artifact of unusually-precise hand-authored titles.** When
> task titles are realistically coarse — as `loom decompose`
> actually produces them — D1d and D2d collapse to **0 % across
> all metrics**, the same shape as R6 D1/D2.
>
> **Refined claim:** Loom's value comes from **both** the chain
> decomposition (so multi-file refactors are tractable) **and**
> the spec-context delivery (so coarse task titles get structural
> shape). Either alone is insufficient at realistic decomposer
> precision. The +100 pp D2 vs D3 isolation is now confirmed at
> the production tier.

## What `loom decompose` actually produces

Auto-decomposer output for the R6 spec:

```yaml
tasks:
- title: Implement RegexField in strings.py
  files_to_modify: [pyschema/fields/strings.py]
  depends_on: []
- title: Re-export RegexField in fields/__init__.py
  files_to_modify: [pyschema/fields/__init__.py]
  depends_on: [Implement RegexField in strings.py]
- title: Re-export RegexField in __init__.py
  files_to_modify: [pyschema/__init__.py]
  depends_on: [Re-export RegexField in fields/__init__.py]
```

vs R6m's hand-authored:

```
T1: "Step 1 of 3: Add the RegexField class to
pyschema/fields/strings.py. Inherits StrField; takes
`pattern: str` (default ''); overrides validate() to call
super().validate() then re.match(self.pattern, result). Use
@dataclass decorator. Place after the existing UUIDField class."
```

The auto-decomposer:
- ✓ correctly decomposes into 3 tasks
- ✓ correctly identifies the dependency chain
- ✓ correctly assigns files_to_modify
- ✗ produces 4–6 word coarse titles instead of 250-char recipes
- ✗ doesn't include sibling/parent files in `context_files`

The structural pieces are right; the prose precision is far below what a human would write.

## Cells

Identical to R6m, but tasks are produced by `loom decompose --apply`:

| cell | tasks generated by | spec context | semantically equivalent to R6m |
|---|---|---|---|
| **D1d** | `loom decompose` | cleared | D1m, but coarser titles |
| **D2d** | `loom decompose` | stored, undelivered | D2m |
| **D3d** | `loom decompose` | delivered to each task | D3m |

Each trial:
1. Set up workspace (pyschema-extended pre-written, gate tests).
2. Seed `Requirement` + `Specification` with the R6 rule text.
3. Run `loom decompose <spec_id> --apply --target-dir <workspace>`.
4. Patch tasks per cell (clear context_specs for D1d/D2d, repoint test_to_write to per-task gate tests).
5. Run `loom_exec --next --loop` to drain the queue.
6. Grade with 5 metrics.

Decomposer model: `ollama:qwen2.5-coder:32b` (the local default).
Executor model: `qwen3.5:latest` (same as all prior smokes).

## What this finally pins down

Across five R6-family experiments (R6 → R6m → phV), the Loom value
proposition has been progressively isolated:

1. **R6 D3 (single coarse task + delivered spec):** 100 % acceptance,
   0 % wiring → exposed the single-task-scope architectural ceiling.
2. **R6m D3m (hand-authored chain + delivered spec):** 100 % across all
   5 metrics → showed the chain mechanism resolves the wiring ceiling.
3. **R6m D1m (hand-authored chain, no spec):** 95 % across all metrics
   → suggested decomposition alone might be load-bearing (later
   recanted).
4. **phV D3d (auto-decomposed chain + delivered spec):** **100 %**
   across all 5 metrics → confirms the chain + spec works at the
   production decomposer's precision tier.
5. **phV D1d/D2d (auto-decomposed chain, no spec):** **0 %** → confirms
   spec delivery is load-bearing when titles are realistically coarse.

The corrected, production-grounded Loom claim:

> **Loom's value is in providing BOTH the chain decomposition
> (so multi-file refactors are tractable) AND the spec-context
> delivery (so coarse task titles get structural shape). With
> the actual `loom decompose` → `loom_exec --loop` pipeline,
> multi-file aligning refactors complete at 100 % across all 5
> metrics on qwen3.5 + Python. Without spec delivery, the same
> pipeline produces 0 %.**

This is the strongest production-claim evidence in the session.

## Why the R6m D1m surprise didn't hold

R6m's hand-authored task titles encoded most of the rule structure
*inside the title string*: file path, base class, decorator, method
override pattern, default value, exact placement instructions. The
title was effectively a 250-char micro-spec.

The actual decomposer (Ollama qwen2.5-coder:32b on the SPEC text)
produces 4–6 word titles that are structurally correct but
qualitatively coarse — which is what we should expect from a model
that's compressing a 1500-char spec into a few task labels.

When titles are this coarse, qwen-the-executor doesn't have enough
context from the title alone to produce idiomatic correct code.
The spec needs to reach the prompt some other way.

That "other way" is exactly what `task_build_prompt` does: when
the task has `context_specs=[spec_id]`, the spec body is injected.
That's why D3d hits 100 %.

## Limitations + follow-ons

- **Single decomposer model** (`ollama:qwen2.5-coder:32b`). Opus-driven
  decomposition (with `ANTHROPIC_API_KEY`) would presumably produce
  more precise titles — possibly closing the D1d gap. Worth testing.
- **Single language (Python).** Cross-language map already showed
  Loom's lift varies. Auto-decompose + chained tasks in TS/Rust/Java
  might shift the picture.
- **The decomposer doesn't include sibling files in context_files.**
  R6m hand-authored T1 included primitives.py, base.py, errors.py
  (so qwen could see Email/URL/UUID as analogies). Auto-decomposer
  output only includes the file being modified. This may be a
  `loom decompose` improvement opportunity. With spec delivered,
  it doesn't matter (D3d hit 100 %); without spec, it might matter.
- **One refactor type (add new class wired through barrels).** Other
  multi-file changes might decompose differently.

### Recommended next experiments (priority order)

1. **Auto-decompose + Opus** — if `ANTHROPIC_API_KEY` is set, does
   Opus-driven decomposition produce R6m-precision titles? If yes,
   the decomposer-quality gap is local-model-only. If no, it's a
   prompt-engineering gap in `loom decompose`.
2. **Real-codebase replay** — pick a small open-source library, run
   the actual production pipeline (loom decompose → exec) on a
   historical PR. Compare to merged code.
3. **Auto-decompose context_files improvement** — tweak the
   decomposer's prompt to include sibling/parent files in
   context_files. Re-run phV D1d/D2d. Test whether richer context
   without spec delivery moves the needle.
4. **Cross-language phV ports** — TS / Rust / Java auto-decompose
   smokes. Are auto-decomposer titles equally coarse across
   languages? Does spec delivery save them similarly?
