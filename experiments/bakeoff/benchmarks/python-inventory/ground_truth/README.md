# python-inventory benchmark

Multi-service Python benchmark — three coordinating services
(customers, inventory, orders) over an in-memory persistence store
with snapshot / restore. Direct port of `dart-inventory` to Python
idioms (kwargs, dataclasses, `__post_init__`) so we can isolate
language effects from complexity effects.

## Public API

The library is exported from `shop/__init__.py` (a barrel — pre-written
by the harness, not generated by the executor). All other shop files
are the executor's responsibility.

```python
from shop import (
    Address, Customer, CustomerService,
    Product, InventoryService,
    Order, OrderStatus, OrderService,
    Store,
)

store = Store()
customers = CustomerService(store)
inventory = InventoryService(store)
orders = OrderService(store, customers, inventory)

customers.register(id="c1", name="Alice", email="alice@example.com")
inventory.register_product(sku="A", name="Widget", price=9.99)
inventory.add_stock("A", 100)

order = orders.place(
    customer_id="c1",
    lines=[{"sku": "A", "quantity": 3}],
)
orders.mark_paid(order.id)
orders.ship(order.id)    # commits reservations, decrements on_hand
orders.deliver(order.id)
```

## Domain model (REQ-1 through REQ-6)

### REQ-1: Error hierarchy (`errors.py`)

All domain errors derive from `DomainError(Exception)`. Subclass
names are exact — tests assert on type:

- `ValidationError` — input did not meet a precondition
- `NotFoundError` — id/sku not in the store
- `ConflictError` — duplicate id/sku
- `InsufficientStockError` — `available < requested`
- `InvalidTransitionError` — status transition not allowed
- `ReservationError` — operation on a closed reservation token

Each subclass body is just `pass` (no extra fields).

### REQ-2: Customer types + service (`types/customers.py`, `services/customer_service.py`)

`Customer` is a `@dataclass` with `id: str`, `name: str`, `email: str`,
`addresses: List[Address] = field(default_factory=list)`. Validates
in `__post_init__`: id non-empty, name non-empty, email contains `@`.
`Address` is a `@dataclass(frozen=True)` value type with `street`,
`city`, `postal_code`.

`CustomerService` exposes `register(*, id, name, email)`,
`get(id)`, `add_address(id, address)`. Backed by `Store.customers`.
Duplicate id on register raises `ConflictError`; unknown id on get
raises `NotFoundError`.

### REQ-3: Product type + stock types (`types/products.py`, `types/inventory.py`)

`Product` is `@dataclass(frozen=True)`: `sku: str`, `name: str`,
`price: float` (must be > 0), validated in `__post_init__`.

`StockLevel` is `@dataclass`: `sku: str`, `on_hand: int = 0`,
`reserved: int = 0`, plus a `@property available` returning
`on_hand - reserved`.

`ReservationToken` is `@dataclass`: `token_id: str`, `order_id: str`,
`sku: str`, `quantity: int`, `committed: bool = False`,
`released: bool = False`, plus `@property is_open` returning
`not committed and not released`.

### REQ-4: Inventory service (`services/inventory_service.py`)

`InventoryService(store)` operates on `store.products`,
`store.stock`, `store.reservations`. Token IDs are formatted
`rsv-NNNNNN` (zero-padded 6-digit, ascending).

- `register_product(*, sku, name, price) -> Product` — also creates
  an empty `StockLevel` for the sku
- `get_product(sku) -> Product`, `stock_of(sku) -> StockLevel`
- `add_stock(sku, qty)` — qty > 0 required
- `reserve(*, order_id, sku, quantity) -> ReservationToken` —
  increments `StockLevel.reserved`. Insufficient available raises
  `InsufficientStockError`.
- `commit(token_id)` — closes token, decrements `on_hand` AND
  `reserved`. On already-closed token, raises `ReservationError`.
- `release(token_id)` — closes token, decrements `reserved`.
  On already-closed token, raises `ReservationError`.

### REQ-5: Order types + service (`types/orders.py`, `services/order_service.py`)

`OrderStatus(Enum)` with values `NEW`, `PAID`, `SHIPPED`, `DELIVERED`,
`CANCELLED`. (Use uppercase enum value names.)

`Item` is `@dataclass(frozen=True)`: `sku: str`, `quantity: int`,
`unit_price: float`. Validates in `__post_init__`. `@property
line_total` returns `quantity * unit_price`. Item snapshots
`unit_price` at order placement so later product price changes do
not retroactively affect totals.

`Transition` is `@dataclass(frozen=True)`: `from_status: Optional[OrderStatus]`
(nullable for the initial creation record), `to_status: OrderStatus`,
`timestamp: datetime`.

`Order` is `@dataclass`: `id: str` (`ord-NNNNNN`),
`customer_id: str`, `items: List[Item]`, `status: OrderStatus =
OrderStatus.NEW`, `history: List[Transition] = field(...)`,
`reservation_tokens: List[str] = field(...)`. Validates in
`__post_init__`. `@property total` aggregates `line_total` over items.

`OrderService(store, customers, inventory)`:

- `place(*, customer_id, lines)` — `lines` is `List[dict]` with keys
  `'sku'` and `'quantity'`. Validates customer + each sku exists,
  then reserves stock for each line. On failure mid-loop, releases
  all already-made reservations from this call and re-raises. On
  success: appends an initial `Transition(from_status=None,
  to_status=OrderStatus.NEW)` to history.
- `get(order_id) -> Order`
- `mark_paid(order_id)` — NEW → PAID; otherwise raises
  `InvalidTransitionError`.
- `ship(order_id)` — PAID → SHIPPED. **Commits** all reservation
  tokens stored on the order before transitioning.
- `deliver(order_id)` — SHIPPED → DELIVERED.
- `cancel(order_id)` — NEW OR PAID → CANCELLED. Releases all open
  reservation tokens. From SHIPPED/DELIVERED/CANCELLED raises
  `InvalidTransitionError`.

### REQ-6: Persistence (`persistence.py`)

`Store` is a `@dataclass` holding five dicts (customers, products,
stock, orders, reservations) all default to empty `dict`. `Snapshot`
is the same shape. `snapshot()` must DEEP-COPY mutable state
(StockLevel, Order including its history list, ReservationToken)
so post-snapshot mutations don't bleed through `restore()`.
Customers and Products are immutable so shallow `dict()` is fine
for them. `restore(snap)` clears all current dicts and replaces
them from the snapshot.

## Why this benchmark

This is the Python sibling of `dart-inventory`. We use it to
disambiguate two hypotheses about the dart-inventory complexity
ceiling:

- **H1 (Dart-specific):** qwen3.5 has Dart blind spots — named-args
  syntax, `const` constructors, records — that wouldn't exist in
  Python. Under H1: python-inventory should pass at non-zero rate.
- **H2 (Complexity-specific):** qwen3.5 hits a coordination ceiling
  at 8+ files / 700+ LOC regardless of language. Under H2:
  python-inventory should fail like dart-inventory.

Same domain semantics, same test count (28), same task count (8) —
the only difference is the executor's language. Result is paper-
relevant either way.
