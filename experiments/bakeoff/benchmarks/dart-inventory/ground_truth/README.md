# dart-inventory benchmark

Multi-service Dart benchmark — three coordinating services (customers,
inventory, orders) over an in-memory persistence store with snapshot /
restore. Larger than `dart-orders` (3 files → 8 files) so multi-file
orchestration improvements have headroom to show lift.

## Public API

The library is exported from `lib/shop.dart` (a barrel — pre-written by
the harness, not generated by the executor). All other lib files are
the executor's responsibility.

```dart
import 'package:shop/shop.dart';

final store = Store();
final customers = CustomerService(store);
final inventory = InventoryService(store);
final orders = OrderService(store, customers, inventory);

customers.register(id: 'c1', name: 'Alice', email: 'alice@example.com');
inventory.registerProduct(sku: 'A', name: 'Widget', price: 9.99);
inventory.addStock('A', 100);

final order = orders.place(
  customerId: 'c1',
  lines: [(sku: 'A', quantity: 3)],
);
orders.markPaid(order.id);
orders.ship(order.id);    // commits reservations, decrements onHand
orders.deliver(order.id);
```

## Files (8 executor tasks + 1 pre-written barrel)

```
lib/
├── shop.dart                       (barrel — pre-written by driver)
├── errors.dart                     [task 1]
├── types/
│   ├── customers.dart              [task 2]
│   ├── products.dart               [task 3]
│   ├── inventory.dart              [task 4]
│   └── orders.dart                 [task 5]
├── persistence.dart                [task 6]
└── services/
    ├── customer_service.dart       [task 7]
    ├── inventory_service.dart      [task 8]
    └── order_service.dart          [task 9]
```

Tasks run in topological order. Each task's executor sees the spec
section labeled `### lib/<path>.dart`.

## Domain model (REQ-1 through REQ-6)

### REQ-1: Error hierarchy (`errors.dart`)

All domain errors extend `DomainError implements Exception` with a
`message` field. The exact subclass names matter — tests assert on type:

- `ValidationError` — input did not meet a precondition
- `NotFoundError` — id/sku not in the store
- `ConflictError` — duplicate id/sku
- `InsufficientStockError` — `available < requested`
- `InvalidTransitionError` — status transition not allowed
- `ReservationError` — operation on a closed reservation token

### REQ-2: Customer types + service (`types/customers.dart`, `services/customer_service.dart`)

`Customer` holds `id`, `name`, `email`, and a mutable `addresses`
list. Constructor validates: id non-empty, name non-empty, email
contains `@`. `Address` is an immutable value type with `street`,
`city`, `postalCode`.

`CustomerService` exposes `register`, `get`, `addAddress`. Backed by
`Store.customers`. Duplicate id on register raises `ConflictError`;
unknown id on get raises `NotFoundError`.

### REQ-3: Product type + stock types (`types/products.dart`, `types/inventory.dart`)

`Product` is immutable: `sku`, `name`, `price` (must be > 0).

`StockLevel` mutable: `onHand`, `reserved`, plus a derived
`available = onHand - reserved`.

`ReservationToken`: `tokenId`, `orderId`, `sku`, `quantity`, plus
mutable `committed`/`released` flags. `isOpen` is `true` iff neither
committed nor released.

### REQ-4: Inventory service (`services/inventory_service.dart`)

`InventoryService(Store)` operates on `Store.products`, `Store.stock`,
`Store.reservations`. Token IDs are formatted `rsv-NNNNNN`
(zero-padded 6-digit, ascending).

- `registerProduct(sku, name, price) -> Product` (also creates an
  empty `StockLevel` for the sku)
- `getProduct(sku) -> Product`, `stockOf(sku) -> StockLevel`
- `addStock(sku, qty)` — qty > 0 required
- `reserve(orderId, sku, quantity) -> ReservationToken` — increments
  `StockLevel.reserved`. Insufficient available raises
  `InsufficientStockError`.
- `commit(tokenId)` — closes token, decrements `onHand` AND
  `reserved`. On already-closed token, raises `ReservationError`.
- `release(tokenId)` — closes token, decrements `reserved` (returns
  to available). On already-closed token, raises `ReservationError`.

### REQ-5: Order types + service (`types/orders.dart`, `services/order_service.dart`)

`OrderStatus` enum: `newly`, `paid`, `shipped`, `delivered`,
`cancelled`. (Note: `newly` instead of `new` because Dart reserves
`new` as a keyword.)

`Item` snapshots `unitPrice` at order placement so later product
price changes do not retroactively affect the order's `total`.

`Transition` records `fromStatus` (nullable for the initial creation
record), `toStatus`, `timestamp`.

`Order` holds: `id` (`ord-NNNNNN`), `customerId`, `items`, mutable
`status` (default `OrderStatus.newly`), `history` list of
`Transition`, `reservationTokens` list of token IDs. Constructor
validates id, customerId, non-empty items.

`OrderService(Store, CustomerService, InventoryService)`:

- `place({customerId, lines: List<({String sku, int quantity})>})
  -> Order` — validates customer + each sku exists, then reserves
  stock for each line. On failure mid-loop, releases all already-made
  reservations from this call and rethrows. On success: appends an
  initial `Transition(fromStatus: null, toStatus: newly)` to history.
- `get(orderId) -> Order`
- `markPaid(orderId)` — newly → paid; otherwise raises
  `InvalidTransitionError`.
- `ship(orderId)` — paid → shipped. **Commits** all reservation
  tokens stored on the order before transitioning.
- `deliver(orderId)` — shipped → delivered.
- `cancel(orderId)` — newly OR paid → cancelled. Releases all open
  reservation tokens. From shipped/delivered/cancelled raises
  `InvalidTransitionError`.

### REQ-6: Persistence (`persistence.dart`)

`Store` holds five maps (customers, products, stock, orders,
reservations). `Snapshot` carries a deep-enough copy. `snapshot()`
must DEEP-COPY mutable state (StockLevel fields, Order status/history,
ReservationToken flags) so post-snapshot mutations don't bleed
through `restore()`. `restore()` clears all current maps and replaces
them from the snapshot.

## Why this benchmark

`dart-orders` (3 files) is small enough that Tier 2 orchestration
improvements (retry-with-error, temperature ramp, escalation) saturate
at 100 % — there's no headroom for a blueprint pass to show lift.

`dart-inventory` (8 files, 4-layer DAG) gives blueprint refinement
room to breathe: errors → types/* → persistence → services/* is a
dependency cascade where one file's wrong abstraction (e.g. swapping
`StockLevel.reserved` for a method, defining `Item.unitPrice` as
`int` instead of `double`, or forgetting `OrderStatus.newly`'s
keyword-collision rename) breaks downstream tasks the executor never
sees in isolation. Compile-error feedback can't repair a wrong
abstraction; a validated blueprint can prevent it.
