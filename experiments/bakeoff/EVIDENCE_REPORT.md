## How to read these tables (per-phase metric notes)

Several phases measure something other than "did the test
suite pass." Reading the `full-pass` column literally for
those phases will mislead.

- **Phase A / B (in-session Â±Loom):** TaskQueue and
  state-machine were *saturated* at every Claude tier. Both
  cells (with and without Loom) hit 100 %. The phase
  measures **cost overhead**, not correctness lift. See
  per-phase `total cost` columns.

- **Phase D (asymmetric pipeline):** Pass-rate matters here
  AND so does cost ratio vs. the in-session baseline. The
  ~8Ă— cost claim comes from comparing D's `total cost` to
  the equivalent Phase B cell's `total cost`. See
  `FINDINGS-bakeoff-v2-pilot.md` for the breakdown.

- **Phase E (pre-edit hook on/off):** "pass" here means
  *the agent complied with the requirement under the
  hook setting*. The headline finding (+93pp Sonnet, +60pp
  Haiku, 0pp Opus) compares the *delta* between hook-on and
  hook-off cells, not the absolute pass rate.

- **Phase E.scale:** measures hook *latency* under 100 /
  500-file synthetic projects, not pass rate. The 0/0
  full-pass count in the table is an artifact of the
  aggregator; the actual finding is constant-time hook
  latency around 800 ms.

- **Phase E.block (hard-block on drift):** "pass=false" is
  the *expected* outcome â€” the hook blocked the edit. The
  0/30 in the table is the agent failing to bypass the
  block, which is exactly what we wanted. The mechanism is
  30/30 reliable; the 0 % "pass rate" is a misreading of
  the field semantics.

- **Phase G (cross-session rationale):** "pass" is whether
  the agent cited the rationale verbatim or paraphrased.
  The headline finding (100 % Haiku, 93 % Sonnet citation
  rate) comes from the with-rationale cell only;
  without-rationale cell stays near 0 %. The aggregate
  62.8 % across all G cells in this table mixes those.
