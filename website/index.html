Explain it like I'm five

The three things Loom does, in plain language.

1

Smart planner + cheap worker = same job, way less money

Imagine you're building a Lego castle. You can pay an expert to do every brick ($$$), or pay them just to draw the blueprint, then let a kid do the actual stacking (free).

Loom does that for AI coding. Have Claude Opus (the expensive model) write a really detailed plan, and have qwen3.5 (a free model on your computer) write the actual code. Same exact answer, ~8× cheaper at parity quality on hidden test suites.

The kid can stack Legos. You just need the expert for the thinking part.

2

A safety nudge before the edit makes the AI follow the rules

Imagine telling a kid "before drawing on any wall, check the card I taped to the door." The card says "we use crayons here, never markers."

Loom does this for AI. Right before the AI edits a file, Loom whispers "hey, this file has this rule: use LegacyConfig.get, not os.environ." The AI reads the whisper and follows the rule.

Without the whisper: weaker AIs get the rule wrong 6 out of 10 times. With the whisper: 10 out of 10. Stronger AIs (Sonnet) jump from 1/15 to 15/15 — a huge swing for cents per edit.

3

Telling the AI why makes it remember the reason, not just the rule

Imagine the rule on the card said "don't open the back door." A kid might still open it — why not?

Now imagine the card said "don't open the back door — the dog will run away, like that one Tuesday last year." Now the kid gets it and won't open the door.

Loom lets you save the why next to every rule. With just the rule, AI follows along about 67% of the time and never mentions why. With rule + reason: 93% follows AND 100% mention the reason back to you — proof it actually understood, not just obeyed.

We even controlled for "more text is more memorable": fake gibberish-padding of the same length doesn't get cited back. Only the real rationale does.

C how_it_works.spec [ ARCH ]

How it works

Four primitives. One workflow.

Capture

Every decision becomes a Requirement with structured metadata: domain, value, rationale, optional acceptance criteria. They're embedded into a local SQLite store keyed to your project (pluggable embedding provider: Ollama, OpenAI, or deterministic hash).

Link

loom link attaches files to the requirements they implement. The store carries content hashes so drift is detectable; line ranges so the link stays specific. last_referenced is stamped passively, so loom stale can surface cold requirements without forced deletion.

Surface

A PreToolUse hook fires before every Edit / Write the agent makes. It consults Loom and injects the linked requirements + their rationale as a system-reminder. Optionally hard-blocks on drift. loom metrics and loom health-score aggregate the resulting event log into a single CI-gateable number.

Decompose & execute

For greenfield work, loom decompose turns a spec into atomic tasks with full context bundles. loom_exec drains them with a small local model — frontier-quality results at local-model cost. Each task runs in a scratch copy and only promotes to the working tree when the grading test passes. 

# install
$ pip install loom-cli

# capture a decision and its why
$ echo "REQ: All config goes through LegacyConfig.get()" | \
    loom extract --rationale "Schema validation; raw env reads bypass it"

# link the file to the requirement
$ loom link app/services.py --req REQ-abc123

# the hook fires automatically before any edit to services.py
# the agent sees the rule + rationale and complies

# effectiveness telemetry, ready for CI gating
$ loom health-score --json | jq .score

E install.sh [ START ]

Get started

One pip install, then loom init in your repo.

Loom v1 ships as a Python package: loom and loom_exec on PATH after install, plus a real import loom for embedding into your own tooling. Pluggable embedding providers (Ollama, OpenAI, deterministic hash), passive staleness tracking, effectiveness metrics, and a single 0–100 health score ready for CI gating. 

# install
$ pip install loom-cli

# pull the embedding model (default Ollama provider)
$ ollama pull nomic-embed-text

# onboard a target repo
$ cd ~/path/to/my-project
$ loom init

# try it
$ echo "REQUIREMENT: behavior | confirm before deletion" | \
    loom extract --rationale "prevent accidental data loss"
$ loom doctor
$ loom health-score --json | jq .score

Step-by-step onboarding with success indicators at every stage: GETTING_STARTED.md.
