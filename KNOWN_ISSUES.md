# Known Issues

Outstanding issues identified during development but not yet fixed. Grouped by area.

Each issue lists: what breaks, why it was deferred, and what the fix looks like. Pick these up as standalone follow-ups.

---

## Hooks (.claude/settings.json)

### H1. `if` filter on Bash hooks doesn't actually gate execution

**Observed:** A PostToolUse hook with `"matcher": "Bash"` and `"if": "Bash(git commit:*)"` fires on every Bash command, not just git commits. The hook spawns a Python process and runs git commands every time Claude Code runs any shell command.

**Why deferred:** Not a correctness bug — the hook's body has a docs-only guard that prevents unwanted sync runs. But it's a performance issue (subprocess overhead on every Bash call).

**Current workaround (committed in `518df23`):** Read the command from stdin JSON and `grep -q 'git commit' || exit 0` at the top of the command. This avoids the Python subprocess but still spawns a shell.

**Real fix:** Investigate whether the permission-rule syntax for `if` should be `Bash(git commit:*)` or `Bash(git commit*)` or something else. The skill docs say `"Bash(git:*)"` is the prefix-wildcard form, but `Bash(git commit:*)` may not parse. Test with `claude --debug` to see what matcher is actually being evaluated.

### H2. Settings watcher doesn't reload mid-session

**Observed:** Edits to `.claude/settings.json` during a session don't take effect until session restart or `/hooks` menu reload.

**Why deferred:** This is a Claude Code platform behavior, not something Loom can fix. Noted so future developers understand why hook changes seem to "not work" until next session.

**Mitigation:** Document this in CLAUDE.md. Tell users to open `/hooks` after editing settings.

### H3. `loom sync` always rewrites the timestamp, causing no-op doc changes

**Observed:** Every run of `loom sync` updates the `*Auto-generated by Loom — <timestamp>*` header even if nothing in the store changed. This creates spurious 2-line diffs in `docs/REQUIREMENTS.md` and `docs/TEST_SPEC.md` after every commit that touches non-docs files, triggering a pointless re-commit cycle.

**Why deferred:** Safe to ignore — the sync hook's docs-only guard prevents infinite loops, and timestamp-only diffs can be discarded with `git checkout -- docs/`.

**Fix:** In `src/docs.py`, compare new content to the existing file (minus timestamp) before writing. If identical, leave the file alone. Or: make the timestamp optional / opt-in via a flag.

## CLI

### C1. `-p/--project` must come before the subcommand

**Observed:** `loom doctor -p myproject` fails with `unrecognized arguments: -p myproject` — the flag must be `loom -p myproject doctor`.

**Why deferred:** argparse's default behavior. Documented in CLAUDE.md examples but not a blocker.

**Fix:** Add `-p/--project` to each subparser as well as the parent parser, so it works in either position. Or restructure with `dest=` to collapse them.

### C2. Shebang warning fallback log message is noisy

**Observed:** When Ollama is unreachable, every CLI call prints `⚠️  Ollama unavailable after 3 attempts, using fallback embeddings: ...` — even on `--json` output, where it goes to stderr but still clutters logs.

**Why deferred:** Cosmetic. The JSON output itself is clean (message goes to stderr).

**Fix:** Suppress the warning when `--json` is set, or only print it once per process.

## Model / Data

### M1. `TestSpec` is still indexed by `req_id`, not its own ID

**Observed:** `TestSpec` objects in `.loom-specs.json` use `req_id` as the dict key — meaning you can only have one test spec per requirement. The new `spec_ids` field on `TestSpec` lets a test declare which specs it verifies, but the underlying store is still req-keyed.

**Why deferred:** Biggest impact would be a migration and a bigger refactor. The three-layer coverage model works fine for now because one test per requirement is typical, and specs are the "how" layer.

**Fix:** Give TestSpec its own ID (e.g., `TS-abc123`), key the JSON store by that, and let multiple test specs per requirement (or per spec) coexist.

### M2. `Implementation.satisfies` is always populated, even when linking to a spec

**Observed:** When you run `loom link file --spec SPEC-xxx`, the implementation records both `satisfies_specs=[SPEC-xxx]` AND `satisfies=[{req_id: parent_req, ...}]` (auto-linked via the spec's parent). This is intentional — it keeps legacy req→impl queries working — but it means `get_implementations_for_requirement(req)` returns the same impl as `get_implementations_for_specification(spec)`.

**Why deferred:** This is by design for now (dual-index), but downstream consumers need to be careful not to double-count when computing coverage metrics.

**Fix (eventual):** Add a flag `direct: bool` on the req link, so queries can distinguish direct req→impl links from auto-propagated spec→impl→req chains.

### M3. No backwards migration for existing stores

**Observed:** Stores created before the `rationale` / `spec_ids` fields exist will load fine (thanks to `setdefault` in `from_dict`), but there's no command to backfill these fields or upgrade stored data.

**Why deferred:** `setdefault` handles read compatibility; no stores exist in the wild yet that need writing migration.

**Fix (if/when needed):** A `loom migrate` command that reads, enriches, and rewrites entries.

## Testing

### T1. CLI tests require the skill to be installed

**Observed:** `tests/test_cli.py` shells out to `~/.openclaw/skills/loom/scripts/loom` — it won't run against the source tree directly.

**Why deferred:** Documented in CLAUDE.md. Requires manual install step.

**Fix:** Make the path configurable via env var (`LOOM_CLI_PATH`) defaulting to the repo's `scripts/loom`.

### T2. No test coverage for `cmd_coverage` or `cmd_link --spec`

**Observed:** The three-layer coverage refactor has no unit tests. Manually verified but not automated.

**Why deferred:** `cmd_*` functions are not unit-testable without more plumbing (they call `print`, `sys.exit`, use `args`). The store and docs functions are tested.

**Fix:** Add subprocess-based tests in `tests/test_cli.py` that exercise `loom coverage --json` and `loom link --spec --json` against a temp project.

### T3. Sample embedding in tests is `[0.1] * 768`

**Observed:** All tests use an identical embedding, which means semantic search tests don't actually test semantic similarity (everything has distance 0).

**Why deferred:** Tests check structural behavior, not similarity quality. Real similarity is exercised via manual testing with Ollama.

**Fix:** Use slightly different embeddings per test requirement so search can actually rank them.

## Documentation

### D1. README.md and SKILL.md don't mention the three-layer model

**Observed:** The refactor made specs central to the workflow, but README.md and SKILL.md still describe the direct req→impl flow.

**Why deferred:** User-facing docs outside of CLAUDE.md and ROADMAP.md weren't the scope of the refactor.

**Fix:** Update README.md's "Usage Patterns" and SKILL.md's "Example Flow" to show the three-layer workflow (`extract` → `spec` → `link --spec` → `test`).

### D2. MCP server skeleton has no handler implementations

**Observed:** `mcp_server/server.py` declares tools and dispatch but all handlers except `loom_query` and `loom_list` raise `NotImplementedError`. `_embed()` is also a stub.

**Why deferred:** Needs one remaining upstream refactor — split `cmd_*` functions so the MCP handlers can share logic with the CLI without reimplementing it. (The embedding extraction — the other prerequisite — is done.) Filling in handlers before the `cmd_*` split would create duplication.

**Fix:** Milestone 4.2 proper. See `mcp_server/README.md` for the sequence.

### D3. `prompts/extract.md` and `prompts/link.md` not updated

**Observed:** Extraction and linking prompt templates may reference the old model.

**Why deferred:** Not reviewed in this session.

**Fix:** Audit both prompt files and update to encourage spec creation after extract, and --spec linking instead of --req.
